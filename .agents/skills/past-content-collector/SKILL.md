# 历史内容采集器

## 概述

本技能负责把公开或已登录可见的账号主页、栏目页、合集页里的历史内容整理成结构化资料。它只做采集和归档，不替代 `knowledge-bootstrap` 的公司画像、产品、用户和关键词整理。

## 触发场景

- 用户提供小红书、公众号、抖音、视频号、B站、知乎、官网博客等主页链接，并希望补充历史内容。
- 初始化知识库时，用户同时给了历史发布账号、主页、合集页或文章列表页。
- 用户要求更新 `knowledge/past-content.md`，并且来源是网页或需要浏览器查看的页面。
- 需要按平台整理已发标题、发布时间、链接、互动数据、内容角度、可复用主题。

## 对用户的话术

面向用户时不要暴露 `knowledge/`、`past-content.md`、`past_content.by_platform` 等内部结构。推荐说：

```text
如果你有小红书、公众号、抖音等历史内容，可以发我主页链接；只要你给主页，我会默认使用你已登录的 Chrome 采集可见历史内容并进行复盘。只有页面实际显示未登录、无权限或内容不可见时，我才会提示你处理登录/授权。
```

## 工具选择

1. 默认采集策略：用户一旦提供小红书、抖音、视频号、公众号、B站、知乎等社媒主页、合集页或内容列表页，就默认假设用户 Chrome 已登录相关平台，并使用 Codex 系统的 Chrome 插件技能 `chrome:control-chrome` 采集页面可见内容；不需要用户额外要求“使用 Chrome”。
2. 官网博客、普通案例页、公开文章列表页可以先用网络请求或普通浏览器读取；如果遇到前端渲染、滚动加载、登录态、反爬提示或内容不可见，立即切到 Chrome。
3. 只有当 Chrome 页面实际显示未登录、无权限、验证拦截或内容不可见时，才暂停采集并请用户在 Chrome 中处理登录/授权后继续；不要绕过平台限制。
4. 不读取浏览器 cookie、local storage、密码、会话文件等敏感信息；只读取页面可见内容。

## 采集边界

- 只记录页面上可见或可合理点击进入后可见的信息。
- 默认用户已在 Chrome 中登录；只有页面实际显示未登录、无权限、验证拦截或内容不可见时，才暂停并请用户处理登录/授权。
- 不绕过平台访问限制，不使用非公开 API，不抓取隐藏接口里的个人隐私。
- 互动数据可能随时间变化，记录采集日期，并标注“采集时可见数据”。
- 无法确认的数据不要补写；用“未显示”“待确认”。

## 采集字段

每个平台至少尽量采集：

- 平台：如小红书、公众号、抖音、视频号、B站、知乎、官网博客
- 账号/栏目名称
- 主页或列表链接
- 内容标题
- 内容链接
- 发布时间或相对时间
- 内容类型：图文、短视频、长视频、文章、直播回放、案例、教程等
- 可见互动数据：阅读、点赞、收藏、评论、转发、播放量
- 内容角度：产品说明、案例、教程、观点、热点借势、客户问题、活动等
- 主题标签或关键词
- 初步表现判断：高互动、普通、缺少数据、无法判断

建议默认采集近 20 条；用户指定数量时按用户要求执行。页面滚动加载时最多尝试到目标数量或明显到底。若可见内容不足 20 条，记录实际可见数量和限制原因。

## 输出到 Facts

供 `knowledge-bootstrap` 消费时，把结果放入 facts 的 `past_content`：

```json
{
  "past_content": {
    "available": true,
    "by_platform": [
      {
        "platform": "小红书",
        "account": "账号名",
        "homepage_url": "https://example.com/account",
        "collected_at": "2026-06-28",
        "items": [
          {
            "title": "内容标题",
            "url": "https://example.com/post",
            "published_at": "2026-06-01",
            "content_type": "图文",
            "metrics": {
              "views": "",
              "likes": "123",
              "comments": "12",
              "favorites": "45",
              "shares": ""
            },
            "angle": "教程",
            "topics": ["关键词 A", "关键词 B"],
            "performance_note": "点赞和收藏相对较高"
          }
        ],
        "observations": [
          "教程类内容互动更集中",
          "标题多使用问题句和场景词"
        ]
      }
    ],
    "items": [],
    "notes": [
      "互动数据为 2026-06-28 页面可见数据",
      "页面可见范围有限，部分数据待确认；仅在页面实际显示未登录/无权限时记录登录限制"
    ]
  }
}
```

`items` 保留为兼容字段；新数据优先写 `by_platform`。如果只拿到零散内容，无法判断平台分组，也可以临时写入 `items`，但要在 `notes` 说明来源限制。

## 写入 `past-content.md`

如果用户只要求更新历史内容文档，而不是完整初始化知识库：

1. 先读取现有 `knowledge/past-content.md`。
2. 保留人工写过的复盘结论。
3. 按平台追加“采集内容列表”“初步观察”“待补充数据”。
4. 不要把“初步观察”写成确定规律；至少需要多平台、多批次或用户确认后，才写“稳定有效模式”。

## 与 `knowledge-bootstrap` 协作

初始化知识库时，如果用户给了账号主页或历史内容链接：

1. 先使用本技能采集历史内容。
2. 把采集结果合并到 `tmp/company-facts.json` 的 `past_content.by_platform`。
3. 再由 `knowledge-bootstrap` 的 `merge_knowledge.py` 统一写入知识库。
4. 如果采集失败，不阻塞其他知识库初始化；在 `source-notes.md` 和 `past-content.md` 写清失败原因、已尝试方式、页面是否实际显示未登录/无权限，以及待补充链接。
